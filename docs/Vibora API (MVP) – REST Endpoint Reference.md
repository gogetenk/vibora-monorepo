# **Vibora API (MVP) – REST Endpoint Reference**

Vibora is a PWA (Progressive Web App) mobile-first matchmaking platform for padel players. This document provides a complete RESTful API reference for the **MVP (Minimum Viable Product)** implementation of Vibora’s backend. The architecture is a modular monolith built with .NET Core 9 using MassTransit for in-process messaging. All endpoints follow standard REST naming conventions (using nouns for resources and standard HTTP verbs for actions[learn.microsoft.com](https://learn.microsoft.com/en-us/azure/architecture/best-practices/api-design#:~:text=,on%20resources)[learn.microsoft.com](https://learn.microsoft.com/en-us/azure/architecture/best-practices/api-design#:~:text=,is%20the%20path%20to%20the)) and communicate using JSON. Each module’s responses are **flat** (non-nested) and optimized as viewmodels to hydrate specific UI screens. Endpoints accessible without authentication (for “play without account” guest usage) are included, and any endpoints planned beyond MVP are marked with **[NON-MVP]**.

## **Users Module**

This module manages user accounts, authentication, profiles, and social features. It includes registration and login, retrieval and update of profiles, and special support for guest users (anonymous play without creating a full account). **Authenticated** endpoints require a valid JWT/auth token in the header, while guest endpoints can be called without authentication (as noted).

| **HTTP Verb** | **Path** | **Description** | **JSON Request Payload** | **JSON Response Payload** |
| --- | --- | --- | --- | --- |
| **POST** | **/users** | Register a new user account with email/password. Returns the created user’s profile and an auth token. *(Public)* | { "name": "string", "email": "string", "password": "string", "skillLevel": "string" }*(e.g. skillLevel could be "Beginner", "Intermediate", etc.)* | { "id": "UUID", "name": "string", "email": "string", "skillLevel": "string", "token": "string" } |
| **POST** | **/users/login** | Authenticate an existing user (login). Returns a JWT auth token and basic profile info on success. *(Public)* | { "email": "string", "password": "string" } | { "id": "UUID", "name": "string", "email": "string", "skillLevel": "string", "token": "string" } |
| **POST** | **/users/guest** | Create a temporary guest user to *“play without an account.”* Returns a guest user profile and token. *(Public)* | { "name": "string", "skillLevel": "string" } | { "id": "UUID", "name": "string", "skillLevel": "string", "token": "string" } |
| **GET** | **/users/me** | Get the current authenticated user’s profile. *(Requires authentication)* | *(None)* | { "id": "UUID", "name": "string", "email": "string", "skillLevel": "string" } |
| **PUT** | **/users/me** | Update the current user’s profile (e.g. change name or skill level). *(Requires authentication)* | { "name": "string", "skillLevel": "string", "email": "string" }*(Fields that are not being changed can be omitted.* Password change may be handled separately.)* | { "id": "UUID", "name": "string", "email": "string", "skillLevel": "string" } |
| **GET** | **/users/{userId}** | Get another player’s public profile by ID (e.g. to view a game partner’s profile). *(Requires authentication)* | *(None)* | { "id": "UUID", "name": "string", "skillLevel": "string", "bio": "string" }*(e.g. bio or other public info; email is omitted for privacy)* |
| **POST** | **/users/{userId}/follow** | **[NON-MVP]** Follow another player. Adds the user to current user’s follow list (for seeing their games or updates). *(Requires authentication)* | *(None)* | { "followedUserId": "UUID", "status": "following" } |
| **DELETE** | **/users/{userId}/follow** | **[NON-MVP]** Unfollow a player that was previously followed. *(Requires authentication)* | *(None)* | { "unfollowedUserId": "UUID", "status": "unfollowed" } |
| **GET** | **/users/me/following** | **[NON-MVP]** Get list of players the current user is following. *(Requires authentication)* | *(None)* | [ { "id": "UUID", "name": "string", "skillLevel": "string" }, … ] |

**Notes:** In the registration and login responses, a **`token`** is provided which must be included in the **`Authorization`** header (**`Bearer token`**) for subsequent authenticated requests. The **`skillLevel`** is a user’s self-reported padel level (e.g., Beginner/Intermediate/Advanced). Guest users created via **`/users/guest`** are intended for one-time use or anonymous play; they receive a token to allow joining games and using chat as a temporary user. Social features like following other players are slated for future releases (non-MVP).

## **Games Module**

The Games module covers creating and managing match listings (“games”), browsing available games, and joining or leaving games. Each game typically represents a padel match that needs players. By default a padel game is doubles (4 players total)[ltapadel.org.uk](https://www.ltapadel.org.uk/play/padel-faqs/#:~:text=Padel%20FAQs%20,padel%20and%20tennis%3F%20Padel), and the API assumes 4 as the maximum players unless specified otherwise. Endpoints exist for public browsing of games (no login needed to view open games) and for both authenticated and anonymous users to join games. Game data is returned in a flat structure optimized for game list or detail screens (including participant info and statuses).

| **HTTP Verb** | **Path** | **Description** | **JSON Request Payload** | **JSON Response Payload** |
| --- | --- | --- | --- | --- |
| **GET** | **/games** | List available upcoming games (matchmaking listings). Supports unauthenticated access for browsing. *(Public)* Optional query filters may be supported (e.g. **`?date=2025-10-10`** or **`?skillLevel=Intermediate`**). | *(None)* | [ { "id": "UUID", "dateTime": "string", "location": "string", "skillLevel": "string", "currentPlayers": 2, "maxPlayers": 4, "hostName": "string" }, … ]*(An array of games with key details; e.g. 2 of 4 players already joined.)* |
| **POST** | **/games** | Create a new game (match) listing. The user creating it becomes the host. *(Requires authentication)* | { "dateTime": "string", "location": "string", "skillLevel": "string", "maxPlayers": 4, "notes": "string" }*(Example: dateTime as ISO 8601 datetime, location name/address, skillLevel required, maxPlayers defaults to 4, optional notes for details.)* | { "id": "UUID", "dateTime": "string", "location": "string", "skillLevel": "string", "maxPlayers": 4, "hostId": "UUID", "currentPlayers": 1, "participants": [ { "id": "UUID", "name": "string", "skillLevel": "string" } ] }*(Returns the new game’s details; the creator is listed as a participant/host.)* |
| **GET** | **/games/{gameId}** | Get full details of a specific game (match). Includes list of participants and status. *(Public – but some fields may be limited if not joined)* | *(None)* | { "id": "UUID", "dateTime": "string", "location": "string", "skillLevel": "string", "maxPlayers": 4, "currentPlayers": 3, "hostId": "UUID", "participants": [ { "id": "UUID", "name": "string", "skillLevel": "string" }, … ], "isFull": false, "isUserJoined": true }*(If request is authenticated, **`isUserJoined`** indicates if the current user (or guest) is one of the participants. **`isFull`** indicates no spots left.)* |
| **POST** | **/games/{gameId}/players** | Join a game as a player. If the request is made by a logged-in user, that user is added to the game. **Unauthenticated users** can also join by providing guest info in the payload. The server will create a guest user and add them as a player. | **Logged-in:** *(No body required)***Guest:** { "name": "string", "skillLevel": "string" } | { "gameId": "UUID", "status": "joined", "participant": { "id": "UUID", "name": "string", "skillLevel": "string" } }*(Confirms the join. For guests, the response includes the generated guest **`id`** so they can be identified in the game.)* |
| **DELETE** | **/games/{gameId}/players/{userId}** | Leave a game (remove a player from the game). The **`{userId}`** path should be the ID of the player leaving. A player can remove themselves (using their own ID, or **`me`** if supported), and the host can remove a participant. *(Requires authentication for known users; guests must use their provided ID and token)* | *(None)* | { "gameId": "UUID", "status": "left", "userId": "UUID" } |
| **DELETE** | **/games/{gameId}** | Cancel a game that the current user is hosting. Marks the game as canceled (or deletes it) and notifies/removes all participants. *(Requires authentication – host only)* | *(None)* | { "gameId": "UUID", "status": "canceled" } |
| **GET** | **/users/me/games** | Get all games the current user is involved in (as host or player). This is used to display “My Games” (upcoming matches for the user). *(Requires authentication)* | *(None)* | [ { "id": "UUID", "dateTime": "string", "location": "string", "skillLevel": "string", "maxPlayers": 4, "currentPlayers": 4, "hostName": "string", "status": "scheduled" }, … ]*(Array of games with key info; could include past or canceled games with status like "completed" or "canceled" if applicable.)* |

**Notes:** When listing games (**`GET /games`**), only games that are open for joining (not full and not started yet) are typically returned by default. The **`dateTime`** is an ISO8601 timestamp for the match start time. The **`location`** is a descriptive name or address of the court/club. The **`participants`** list in game details includes basic info for each joined player. If a game is full (**`currentPlayers == maxPlayers`**), further join attempts will be rejected. Guests joining a game will be treated as a new temporary user in the participants list. Canceling a game is usually only allowed for the host; it will prevent further joins and [in future could notify players or close related chat]. The **My Games** endpoint helps a logged-in user retrieve all matches they are scheduled to play (for showing in their dashboard or profile).

## **Chat Module**

The Chat module enables communication between players in a game. For each game, a dedicated chat channel is available for participants (created automatically when the game is created or when someone joins). The chat endpoints allow fetching the conversation and sending messages. Only players who have joined the game (including guests who joined) can access that game’s chat. Chat messages are delivered in real-time via the PWA (e.g., using websockets in the client), but the REST endpoints below provide fallback or initial load of messages. All chat responses are flattened for easy display (each message as an object with basic fields, no nested user objects).

| **HTTP Verb** | **Path** | **Description** | **JSON Request Payload** | **JSON Response Payload** |
| --- | --- | --- | --- | --- |
| **GET** | **/games/{gameId}/messages** | Retrieve the list of chat messages for the specified game. Returns recent messages in chronological order. *(Requires authentication if the game is private; for open games, guests with the game’s token can also call this)* | *(None)* | [ { "messageId": "UUID", "senderId": "UUID", "senderName": "string", "text": "string", "timestamp": "string" }, … ]*(An array of message objects. **`senderName`** is included for display so the client doesn’t need to lookup user info.)* |
| **POST** | **/games/{gameId}/messages** | Send a new chat message in the game’s chat. The sender must be a participant of the game. *(Requires authentication or guest token)* | { "text": "string" } | { "messageId": "UUID", "senderId": "UUID", "senderName": "string", "text": "string", "timestamp": "string" }*(The created message is echoed back. Timestamp is in ISO8601 format.)* |
| **DELETE** | **/games/{gameId}/chat** | **[NON-MVP]** Close the chat for a game. This would archive or disable new messages in the game’s chat (for example, automatically done some time after the game ends). Typically not exposed to regular users in MVP. *(Admin or system action)* | *(None)* | { "gameId": "UUID", "chatStatus": "closed" } |

**Notes:** Chat message payloads include the sender’s basic identification (both **`senderId`** and a display **`senderName`**) flattened for direct use in the UI, so the app can show messages like “John: [text]” without extra profile lookups. In MVP, chats remain open indefinitely (players can continue to communicate), and there is no automatic deletion; however, a **[NON-MVP]** feature to auto-close chats a certain time after the game ends is planned (hence the placeholder endpoint for closing a chat). All chat endpoints require that the caller has joined the game (either with a user account or as the guest created when joining anonymously); unauthorized access to a game’s chat will be rejected.
